# SOUsers app

iOS application that fetches a list of Stack Overflow users and displays them as a list. No pagination.

The implementation is split into separate layers, bound together via dependency injection. From top to bottom:

- **UI** - `UIViewController` subclasses and custom UI controls.
- **Business logic** - `ViewModel` instances plus utility/helper classes.
- **State** - `AppState` is the core component here - source of truth.
- **Network** - `ApiClient` - abstracts the back-end/data supplier. 

## Tech Stack & Requirements

- Swift 6 (_strict concurrency_)
- UIKit (_SwiftUI included just for #Preview_)
- async/await, `@Observable`, `withObservationTracking`
- MVVM + Coordinator
- Swift Testing
- Xcode 26.3+, iOS 26.1+

Use `SOUsers.xcodeproj` in order to open sources in XCode. Or from the terminal:

```bash
git clone git@github.com:solocez/SOUsers.git && cd SOUsers && xed .
``` 

## Sources organisation

## App

Application entry point.

## Coordinators

Coordinator pattern related sources. `AppCoordinator` is root app coordinator.

## Core

Contains core components' implementation.

### ApiClient

A typed wrapper around the Stack Exchange `/2.3/users` endpoint. Ideally this would be generated by Swagger or swift-openapi-generator. 

### Dependency

A lightweight implementation for Dependency injection. Ideally i'd reach for PointFree's Dependency library, or FactoryKit Injection.

### Models

Keeps common data types used across the app. 

### Persistence

Logic to persist app's data between sessions. Live - persists to UserDefaults. Test - persists into memory.

### State

Contains State of the app - `AppState`. Keeps list of Users. `AppState` is marked with `@Observable` - so ViewController(s) can observe the state directly.

## Screens

Contains Screens of the app. A screen is a couple - `UIViewController` and it's `@Observable` ViewModel class.

### Shared

Commonly used routines.

### UsersList

Shows all fetched Users.

### UserDetails

Show details of a particular user.

## Tests

Folder `SOUsersTests` contains Unit Tests. 

View types are not covered with unit tests. They should be exercised via snapshot tests — PointFree's `swift-snapshot-testing` is the good choice.
